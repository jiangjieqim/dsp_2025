# ROM

    ROM Compiler Directive
    Abbreviation None.

    Arguments (SMALL), (COMPACT), (LARGE), (D512K), or (D16M)

    Default ROM (LARGE)

    µVision Options — Target — Code ROM Size.

    Description You use the ROM directive to specify the size of the program memory. This directive affects the coding of the JMP and CALL instructions.

    +---------------------------------------------------
    |它最直接的影响就是：决定函数调用（CALL）和跳转（JMP）指令是使用 ACALL/AJMP（2字节）还是 LCALL/LJMP（3字节）。
    +---------------------------------------------------

    Option Description 
    SMALL CALL and JMP instructions are coded as ACALL and AJMP. The maximum program size may be 2 KBytes. The entire program must be allocated within the 2 KByte program memory space. 

    +--------------------------------------------------
    |小型CALL和 JMP 指令分别用 ACALL 和 AJMP 表示。程序的最大大小可为 2 KB。整个程序必须分配在2 KB的程序内存空间内。
    +--------------------------------------------------

    COMPACT CALL instructions are coded as LCALL. JMP instructions are coded as AJMP within a function. The size of a function must not exceed 2 KBytes. The entire program may, however, comprise a maximum of 64 KBytes. The type of application determines whether or not ROM (COMPACT) is more advantageous than ROM (LARGE). Any code space saving advantages in using ROM (COMPACT) must be empirically determined. 

    +---------------------
    紧凑型CALL指令以 LCALL 形式编码。JMP 指令在函数内部以 AJMP 形式编码。函数的大小不得超过 2 KB。然而，整个程序的最大容量不得超过64 KB。应用程序的类型决定了紧凑型ROM是否比大容量ROM更具优势。使用ROM（紧凑型）所获得的任何代码空间节省优势均需通过实证方法确定。 
    +---------------------

    LARGE CALL and JMP instructions are coded as LCALL and LJMP. This allows you to use the entire address space without any restrictions. Program size is limited to 64 KBytes. Function size is also limited to 64 KBytes. 
    +----------------------------
    大容量的CALL和 JMP 指令分别编码为 LCALL 和 LJMP 。这使您能够不受任何限制地使用整个地址空间。程序大小限制为 64 KB。函数大小也限制在 64 KB。
    +----------------------------


    D512K†
    (Dallas 390& variants) 19-bit ACALL and AJMP instructions are generated. The maximum program size may be 512 KBytes. This mode is available only for the Dallas 390 and compatible devices. 
    D16M†
    (Dallas 390& variants) 24-bit LCALL instructions and 19-bit AJMP instructions are generated. The maximum program size may be 16MBytes. This mode is available only for the Dallas 390 and compatible devices. 

    Note

    The D512K and D16M options require the OMF2 directive. 
    
    See Also COMPACT, LARGE, SMALL
    
    Example C51 SAMPLE.C ROM (SMALL)

    #pragma ROM (SMALL)
 



    +---------------------------------------------------
    ROM 模式 CALL指令	     JMP 指令	    最大程序大小  适用器件
    SMALL	ACALL	         AJMP	        2 KB	    标准 8051
    COMPACT	LCALL	         AJMP (函数内)  64 KB	    标准 8051
    LARGE	LCALL	         LJMP	        64 KB	    标准 8051
    D512K	19-bitACALL/AJMP 19-bit	        512 KB	    Dallas 390 系列
    D16M	24-bitLCALL	     19-bitAJMP	    16 MB	    Dallas 390 系列
    +--------------------------------------------------

# BROWSE

BROWSE 参数详解
BROWSE 是 Keil C51 编译器的一个编译选项，用于生成浏览信息

    对输出文件的影响
    项目	      未使用 BROWSE	    使用 BROWSE
    .obj文件大小  小	        增大（包含额外信息）
    .lst文件	  基本汇编列表	包含详细符号信息
    .m51 映射文件 基本内存映射  包含符号地址表
    编译时间	  快	            稍慢


    BROWSE 只影响调试/开发体验
    不会改变生成的机器码
    发布版本可以去掉 BROWSE


# DEBUG

    项目	        未使用 DEBUG	使用 DEBUG
    .obj 文件大小	小	            显著增大（2-5倍）
    .OMF 文件	    不含符号表	    包含完整的 OMF-51 符号表
    .lst 文件	    基本汇编	    包含 C 源码行对照
    链接后代码	     无影响	        无影响（只生成符号表）

# OBJECTEXTEND

    简单说：OBJECTEXTEND 是 DEBUG 的增强版，让你在调试时能看到更详细的变量类型信息（特别是结构体），但对仿真器的兼容性有要求。你的配置完全正确，建议在开发阶段保留。

# 优化级别速查表
```c
#pragma OT(0)           /* 关闭优化，最可预测 */
void delay_us(unsigned int us)
{
    while(us--)
    {
        _nop_();        /* 插入空指令精确控制时间 */
        _nop_();
    }
}

#pragma OT(4)           /* 恢复高优化给其他代码 */
void other_function(void)
{
    /* 这里的代码可以使用高优化 */
}
```
```
Keil C51 提供 0-11 共 12 个优化级别，每个高级别包含所有低级别的优化：

级别	核心优化内容	对延时函数的影响
0	常数合并、跳转优化、简单访问优化	最可预测，循环按原样编译，最适合精确延时
1	死代码删除、跳转否定	可能删除“看起来没用”的代码
2	数据覆盖（变量共享内存）	无直接影响
3	窥孔优化（删除冗余MOV指令）	可能减少循环中的指令数
4	寄存器变量、局部公共子表达式删除	循环变量可能被优化到寄存器，大幅加快循环速度
5	全局公共子表达式删除	同上
6	循环旋转	改变循环结构，显著影响延时
7	扩展索引访问优化（使用DPTR）	可能引入额外指令
8	公共尾部合并	多个函数调用共享代码，严重干扰单次延时计算
9	公共块子程序（重复代码转子程序）	延时循环可能被替换为子程序调用
10	重排代码（链接器级）	代码位置改变
11	重用公共退出代码（链接器级）	最极致的大小优化
```
